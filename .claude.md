# Claude Code Rules - LeagueFindr

## General Conventions

### Commit Messages
- Format: `type: short description (50 chars max)`
- Types: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`
- Include detailed explanation in body if changes are non-trivial

### Branching
- `main` - production-ready code
- `dev` - integration branch for features
- `feature/*` - new features
- `fix/*` - bug fixes
- `refactor/*` - code improvements without functionality changes
- `test/*` - testing branches

### Error Handling
- Always wrap errors with context using `fmt.Errorf("context: %w", err)`
- Never silently ignore errors (log or return them)
- For non-critical operations, log error but don't fail (note in comments)

### Logging
- Use `slog` package with structured logging
- Log levels: `Debug` (development info), `Info` (important events), `Warn` (recoverable issues), `Error` (failures)
- Always include relevant context (IDs, operation names, error details)

### Project Structure
```
/backend     - Go API server
/frontend    - Next.js dashboard
/docs        - Documentation
```

### Code Style
- Prefer clarity over brevity
- Add comments explaining "why" not "what"
- Keep functions focused on single responsibility
- Use meaningful variable names (no `a`, `x`, etc.)

### Performance
- Avoid unnecessary API calls
- Cache when possible
- Use context with timeouts for external services
- Profile before optimizing
